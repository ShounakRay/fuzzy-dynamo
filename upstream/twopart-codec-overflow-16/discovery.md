# Discovery: TwoPartCodec Integer Overflow

> **Status: FIXED** -- Filed as upstream issue #6955, fix merged in PR #6959.

## What's the bug?

Computers represent numbers using a fixed number of bits. A 64-bit unsigned integer (`u64` in Rust) can hold values from 0 to 18,446,744,073,709,551,615. What happens if you try to store a number larger than that? The value "overflows" -- it wraps around to the beginning, like a car's odometer rolling from 999,999 back to 000,000. Adding 1 to the maximum value gives you 0, and adding two large numbers can give you a deceptively small result.

The `TwoPartCodec` is a network protocol decoder used in Dynamo's internal communication. When a message arrives over the network, the first 24 bytes are a header containing three 64-bit values: `header_len`, `body_len`, and a checksum. The decoder reads these values, then computes the total expected message size as `24 + header_len + body_len`. This total is compared against a `max_message_size` safety limit to reject unreasonably large messages before allocating memory for them.

The bug is that the addition `24 + header_len + body_len` uses unchecked arithmetic. If an attacker sends a crafted 24-byte message where `header_len` and `body_len` are both near the maximum u64 value, their sum overflows and wraps to a small number. For example, `usize::MAX + usize::MAX + 24` wraps to 22. This tiny result passes the `max_message_size` check (since 22 is less than any reasonable limit), completely defeating the safety guard. In Rust's debug mode, this causes an immediate panic ("attempt to add with overflow"). In release mode, the overflow is silent and the decoder proceeds with a nonsensical total length, potentially reading garbage data from a tiny buffer.

This is a denial-of-service vulnerability: any network-connected attacker can crash (debug mode) or corrupt (release mode) a Dynamo inference server by sending just 24 bytes of crafted data. No authentication is required -- the overflow happens during the earliest stage of message decoding, before any validation of message contents.

## When does this happen in real life?

This bug (now fixed) allowed a network-facing denial-of-service attack:

- **Crafted network input** — any client that can send TCP data to a Dynamo service using `TwoPartCodec` could crash the server by sending 24 bytes of crafted data. The attacker doesn't need valid authentication or a real request — just the ability to open a TCP connection to the service port
- **Internal service communication** — Dynamo services communicate with each other using `TwoPartCodec`. A compromised or buggy worker could crash any service it connects to by sending an overflow-triggering message
- **Debug vs. release divergence** — in debug builds, the overflow panics immediately (crash). In release builds, the overflow wraps silently, causing the size check to pass when it shouldn't. This could potentially lead to reading from or writing to incorrect memory positions

This bug was filed as #6955 and fixed in PR #6959.

## How we found it

### The fuzzing approach

The fuzz target `fuzz_network_codecs.rs` in `lib/runtime/fuzz` uses a **crash oracle** strategy applied to all network-facing codec decoders. It takes raw fuzz-generated bytes and feeds them to every codec in the system: `TcpRequestCodec`, `TcpResponseCodec`, `TcpResponseMessage`, `MsgpackCodec`, and `TwoPartCodec`. The oracle is simple: every codec must return either `Ok` or `Err` on arbitrary input, never panic. The target also tests different `max_message_size` limits (none, 1KB, 1 byte, 24 bytes, 1MB) and streaming scenarios where bytes arrive one at a time.

After the overflow bug was found, a known-bug filter was added to the target: it pre-checks whether the first 16 bytes (as two u64 values) would overflow when added together with 24. Inputs that would overflow are skipped so the fuzzer can continue finding other bugs without repeatedly hitting the same known crash.

### What the fuzzer did

The fuzzer generated a 24-byte input where bytes 8 through 15 contained a near-maximum u64 value. When the `TwoPartCodec` read these bytes, it computed `24 + header_len + body_len`, which overflowed. In debug mode, Rust's built-in overflow checking detected the problem and panicked with "attempt to add with overflow" at `two_part.rs:58`. The fuzzer caught the panic and saved the crashing input as an artifact.

Multiple crash artifacts were found, all sharing the same pattern: the two u64 values in the header summed to more than `usize::MAX - 24`.

### Why traditional testing missed this

Unit tests for the codec used realistic message sizes (kilobytes to megabytes). Nobody constructed a test with near-maximum u64 values in the length fields because those values are nonsensical for real messages -- but an attacker does not send real messages.

## The fix

Replace unchecked addition with checked arithmetic: `24usize.checked_add(header_len).and_then(|v| v.checked_add(body_len))`. If any addition overflows, the decoder returns an error instead of panicking or wrapping. Fixed in upstream PR #6959.

## Fuzzing technique

**Strategy:** Crash oracle with known-bug filtering
**Target:** `fuzz_network_codecs.rs`
**Crate:** `lib/runtime/fuzz`
**Run command:** `cd lib/runtime/fuzz && ~/.cargo/bin/cargo +nightly fuzz run fuzz_network_codecs -- -max_total_time=60`
